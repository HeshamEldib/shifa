دليل المساهمة في مشروع "شفاء"

مرحباً بك في الفريق! لضمان سير العمل بسلاسة والحفاظ على نظافة الكود، نتبع استراتيجية الفروع التالية.

استراتيجية الفروع (Branching Strategy)

نحن نستخدم نموذج GitFlow المبسط.

1. الفروع الرئيسية

main: هذا الفرع يمثل النسخة المستقرة والنهائية من المشروع (الإنتاج). يمنع منعاً باتاً الدفع (Push) إليه مباشرة.

develop: هذا هو فرع التطوير الرئيسي. يجب أن يحتوي دائماً على آخر نسخة تم تطويرها من المشروع. يتم إنشاء جميع فروع الميزات منه.

2. فروع الميزات (Feature Branches)

ابدأ دائماً من develop: قبل البدء في أي ميزة جديدة، تأكد من أنك على فرع develop وقم بسحب آخر التحديثات:

git checkout develop
git pull


أنشئ فرعاً جديداً: قم بإنشاء فرع جديد باسم وصفي للميزة التي تعمل عليها، باستخدام البادئة المناسبة:

للميزات الجديدة: feature/ (مثال: feature/react-booking-page أو feature/dotnet-booking-api)

لإصلاح الأخطاء: fix/ (مثال: fix/login-bug)

# مثال لإنشاء فرع ميزة
git checkout -b feature/react-login-page


اعمل على ميزتك: قم بإجراء التغييرات الخاصة بك وقم بعمل commit بشكل دوري.

3. دمج العمل (Pull Requests - PRs)

الدفع (Push) لفرعك: عندما تنتهي من ميزتك، قم بدفعها إلى المستودع على GitHub:

git push origin feature/react-login-page


إنشاء Pull Request (PR):

اذهب إلى صفحة المستودع على GitHub.

ستجد زرًا لإنشاء "Pull Request" جديد من فرع الميزة الخاص بك.

الهدف (Base): يجب أن يكون دائماً فرع develop.

المصدر (Compare): يجب أن يكون فرع الميزة الخاص بك (مثل feature/react-login-page).

مراجعة الكود (Code Review):

قم بطلب مراجعة من عضو آخر في الفريق (على الأقل مراجع واحد).

لا تقم بدمج الـ PR الخاص بك بنفسك (إلا إذا كان إصلاحًا طارئًا جدًا).

بعد الموافقة على المراجعة، يتم دمج (Merge) الـ PR إلى فرع develop.
